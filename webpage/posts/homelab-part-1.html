Homelab Part 1: Base Setup, Installing Proxmox, and Getting Backups Running

The unglamorous first step every homelab needs — before containers, before Kubernetes, before anything fun.

Every homelab blog starts with the cool stuff — Kubernetes dashboards, automation, self-hosted everything. Nobody writes about step one: getting hypervisors installed, configured properly, and backed up. So that's exactly what this post covers. Nothing more.

The Hardware

My cluster runs on two Dell Precision 3561 workstation laptops:

  • pve-01 — 192.168.1.100
  • pve-02 — 192.168.1.101

Laptops as servers sound wrong until you think about it: built-in UPS (the battery), low power draw, silent, and compact. For a two-node learning cluster, they're ideal.

Installing Proxmox VE

Installation itself is the easy part — flash the Proxmox VE ISO to USB, boot, and follow the installer. The decisions that matter come during and right after:

1. Filesystem. I went with the default ext4 on LVM-thin. ZFS is tempting, but on laptop hardware with limited RAM to spare for VMs, LVM-thin is the pragmatic choice. Thin provisioning means I can over-allocate disk space across VMs without committing it upfront.

2. Static IPs from day one. Both nodes got fixed addresses (.100 and .101) in the installer. A hypervisor that changes IP is a hypervisor you can't find. Set it once, write it down, never touch it again.

3. Hostname and FQDN. pve-01 and pve-02 — short, numbered, boring. Naming schemes that seem clever today are annoying in six months.

Post-Install Base Config

Fresh Proxmox needs a few things before it's actually usable:

Repositories. Out of the box, Proxmox points at the enterprise repo, which fails without a subscription. Switch to the no-subscription repo:

# Disable enterprise repo, enable no-subscription
sed -i "s/^deb/#deb/" /etc/apt/sources.list.d/pve-enterprise.list
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" \
  > /etc/apt/sources.list.d/pve-no-subscription.list
apt update && apt full-upgrade -y

Updates first, everything else second. Run a full upgrade and reboot before creating a single VM. Migrating VMs around later just to patch the host is avoidable pain.

Cluster creation. With both nodes installed and updated, forming the cluster takes two commands — pvecm create on the first node, pvecm add on the second. After that, one web UI manages both machines.

The quorum problem. A two-node cluster has a built-in flaw: if one node dies, the survivor has no majority and refuses to do cluster operations. The fix is a QDevice — a third vote that doesn't need to be a full Proxmox node. In my case, a Raspberry Pi 3 runs corosync-qnetd and acts as the tiebreaker. It does nothing except vote, and that's enough to keep the cluster functional when a node is down.

# On the cluster, after setting up the Pi:
pvecm qdevice setup <pi-ip>

Backups: Proxmox Backup Server

A homelab without backups is a time bomb with a web UI. Before deploying anything I'd miss, I set up Proxmox Backup Server (PBS) — running as a VM (VMID 110) on the cluster itself.

Yes, running the backup server on the infrastructure it backs up is a compromise. The important part: the actual backup datastore lives on separate storage, not on the cluster's own disks. If a node dies, the backups survive; PBS itself can be reinstalled in twenty minutes.

Why PBS instead of plain vzdump backups to a directory?

  • Deduplication — dozens of backup snapshots take a fraction of the raw space, because unchanged chunks are stored once
  • Incremental backups — after the first full backup, only changed blocks transfer. Nightly backups finish in minutes
  • Verification jobs — PBS periodically re-checks stored chunks against their checksums, so I find corruption before I need a restore
  • Pruning policies — keep-last, keep-daily, keep-weekly configured once, enforced automatically

Hooking it up is straightforward: add PBS as storage in the Proxmox datacenter, create a backup job covering all VMs, schedule it nightly, done.

The rule I follow: no workload exists until it's covered by the backup job. New VM, new container — it goes into the backup schedule the same day it's created.

Where This Leaves Things

At the end of step one, the foundation is:

  • Two Proxmox nodes, patched, static IPs, clustered
  • A QDevice keeping quorum honest
  • PBS running nightly deduplicated backups with verification

No services yet. No containers. Nothing exciting. But everything that comes next — and there's a lot coming — sits on top of this, and gets backed up from day one.

That's the whole point of step one: make the boring layer solid, so you never have to think about it again.

Homelab Teil 1: Grundsetup, Proxmox-Installation und laufende Backups

Der unglamouröse erste Schritt, den jedes Homelab braucht — vor Containern, vor Kubernetes, vor allem Spaß.

Jeder Homelab-Blog beginnt mit den coolen Sachen — Kubernetes-Dashboards, Automatisierung, alles self-hosted. Niemand schreibt über Schritt eins: Hypervisoren installieren, sauber konfigurieren und sichern. Genau darum geht es in diesem Beitrag. Um nichts anderes.

Die Hardware

Mein Cluster läuft auf zwei Dell Precision 3561 Workstation-Laptops:

  • pve-01 — 192.168.1.100
  • pve-02 — 192.168.1.101

Laptops als Server klingen falsch, bis man darüber nachdenkt: eingebaute USV (der Akku), geringer Stromverbrauch, lautlos und kompakt. Für einen Zwei-Knoten-Lerncluster sind sie ideal.

Proxmox VE installieren

Die Installation selbst ist der einfache Teil — Proxmox-VE-ISO auf USB flashen, booten, dem Installer folgen. Die Entscheidungen, die zählen, kommen währenddessen und direkt danach:

1. Dateisystem. Ich habe das Standard-ext4 auf LVM-thin gewählt. ZFS ist verlockend, aber auf Laptop-Hardware mit knappem RAM für VMs ist LVM-thin die pragmatische Wahl. Thin Provisioning heißt, ich kann Speicherplatz über VMs hinweg überbuchen, ohne ihn vorab festzulegen.

2. Statische IPs ab Tag eins. Beide Knoten bekamen im Installer feste Adressen (.100 und .101). Ein Hypervisor, der die IP wechselt, ist ein Hypervisor, den man nicht findet. Einmal setzen, aufschreiben, nie wieder anfassen.

3. Hostname und FQDN. pve-01 und pve-02 — kurz, nummeriert, langweilig. Namensschemata, die heute clever wirken, nerven in sechs Monaten.

Basiskonfiguration nach der Installation

Ein frisches Proxmox braucht ein paar Dinge, bevor es wirklich nutzbar ist:

Repositories. Ab Werk zeigt Proxmox auf das Enterprise-Repo, das ohne Subscription fehlschlägt. Also auf das No-Subscription-Repo umstellen:

# Enterprise-Repo deaktivieren, No-Subscription aktivieren
sed -i "s/^deb/#deb/" /etc/apt/sources.list.d/pve-enterprise.list
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" \
  > /etc/apt/sources.list.d/pve-no-subscription.list
apt update && apt full-upgrade -y

Erst Updates, dann alles andere. Vollständiges Upgrade fahren und neu starten, bevor auch nur eine VM erstellt wird. VMs später herumzuschieben, nur um den Host zu patchen, ist vermeidbarer Schmerz.

Cluster erstellen. Sind beide Knoten installiert und aktuell, braucht der Cluster zwei Befehle — pvecm create auf dem ersten Knoten, pvecm add auf dem zweiten. Danach verwaltet eine Web-UI beide Maschinen.

Das Quorum-Problem. Ein Zwei-Knoten-Cluster hat einen eingebauten Makel: Stirbt ein Knoten, hat der Überlebende keine Mehrheit und verweigert Cluster-Operationen. Die Lösung ist ein QDevice — eine dritte Stimme, die kein vollwertiger Proxmox-Knoten sein muss. Bei mir läuft corosync-qnetd auf einem Raspberry Pi 3, der als Zünglein an der Waage dient. Er tut nichts außer abstimmen — und das reicht, damit der Cluster funktionsfähig bleibt, wenn ein Knoten ausfällt.

# Auf dem Cluster, nachdem der Pi eingerichtet ist:
pvecm qdevice setup <pi-ip>

Backups: Proxmox Backup Server

Ein Homelab ohne Backups ist eine Zeitbombe mit Web-UI. Bevor ich irgendetwas deploye, das ich vermissen würde, habe ich den Proxmox Backup Server (PBS) aufgesetzt — als VM (VMID 110) auf dem Cluster selbst.

Ja, den Backup-Server auf der Infrastruktur zu betreiben, die er sichert, ist ein Kompromiss. Entscheidend: Der eigentliche Datastore liegt auf separatem Speicher, nicht auf den Disks des Clusters. Stirbt ein Knoten, überleben die Backups; PBS selbst ist in zwanzig Minuten neu installiert.

Warum PBS statt schlichter vzdump-Backups in ein Verzeichnis?

  • Deduplizierung — Dutzende Backup-Snapshots belegen einen Bruchteil des Rohplatzes, weil unveränderte Chunks nur einmal gespeichert werden
  • Inkrementelle Backups — nach dem ersten Voll-Backup werden nur geänderte Blöcke übertragen. Nächtliche Backups sind in Minuten fertig
  • Verifikationsjobs — PBS prüft gespeicherte Chunks regelmäßig gegen ihre Prüfsummen; Korruption fällt auf, bevor ein Restore nötig ist
  • Pruning-Richtlinien — keep-last, keep-daily, keep-weekly einmal konfiguriert, automatisch durchgesetzt

Das Anbinden ist unkompliziert: PBS als Storage im Proxmox-Datacenter hinzufügen, einen Backup-Job über alle VMs anlegen, nächtlich einplanen, fertig.

Meine Regel: Kein Workload existiert, bevor er vom Backup-Job abgedeckt ist. Neue VM, neuer Container — kommt am selben Tag in den Backup-Zeitplan.

Wo das hinführt

Am Ende von Schritt eins steht das Fundament:

  • Zwei Proxmox-Knoten, gepatcht, statische IPs, geclustert
  • Ein QDevice, das das Quorum ehrlich hält
  • PBS mit nächtlichen deduplizierten Backups inklusive Verifikation

Noch keine Dienste. Keine Container. Nichts Aufregendes. Aber alles, was als Nächstes kommt — und da kommt viel — sitzt auf dieser Basis und wird ab Tag eins gesichert.

Genau das ist der Sinn von Schritt eins: die langweilige Schicht solide bauen, damit man nie wieder über sie nachdenken muss.

Homelab część 1: Konfiguracja bazowa, instalacja Proxmoxa i działające backupy

Mało efektowny pierwszy krok, którego potrzebuje każdy homelab — przed kontenerami, przed Kubernetesem, przed całą zabawą.

Każdy blog o homelabie zaczyna się od fajnych rzeczy — dashboardy Kubernetesa, automatyzacja, wszystko self-hosted. Nikt nie pisze o kroku pierwszym: zainstalowaniu hipernadzorców, porządnej konfiguracji i backupach. I dokładnie o tym jest ten wpis. O niczym więcej.

Sprzęt

Mój klaster działa na dwóch laptopach workstation Dell Precision 3561:

  • pve-01 — 192.168.1.100
  • pve-02 — 192.168.1.101

Laptopy jako serwery brzmią źle, dopóki się nad tym nie zastanowisz: wbudowany UPS (bateria), niski pobór prądu, cisza i kompaktowość. Do dwuwęzłowego klastra do nauki są idealne.

Instalacja Proxmox VE

Sama instalacja to łatwa część — wgraj ISO Proxmox VE na USB, uruchom i podążaj za instalatorem. Decyzje, które się liczą, zapadają w trakcie i tuż po niej:

1. System plików. Wybrałem domyślny ext4 na LVM-thin. ZFS kusi, ale na sprzęcie laptopowym z ograniczonym RAM-em dla VM-ek LVM-thin to pragmatyczny wybór. Thin provisioning oznacza, że mogę nadprzydzielać przestrzeń dyskową między VM-kami bez rezerwowania jej z góry.

2. Statyczne IP od pierwszego dnia. Oba węzły dostały stałe adresy (.100 i .101) już w instalatorze. Hipernadzorca, który zmienia IP, to hipernadzorca, którego nie można znaleźć. Ustaw raz, zapisz, nie ruszaj nigdy więcej.

3. Hostname i FQDN. pve-01 i pve-02 — krótko, numerycznie, nudno. Schematy nazewnictwa, które dziś wydają się sprytne, za pół roku irytują.

Konfiguracja bazowa po instalacji

Świeży Proxmox potrzebuje kilku rzeczy, zanim będzie naprawdę używalny:

Repozytoria. Fabrycznie Proxmox wskazuje na repozytorium enterprise, które bez subskrypcji nie działa. Przełącz na repozytorium no-subscription:

# Wyłącz repo enterprise, włącz no-subscription
sed -i "s/^deb/#deb/" /etc/apt/sources.list.d/pve-enterprise.list
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" \
  > /etc/apt/sources.list.d/pve-no-subscription.list
apt update && apt full-upgrade -y

Najpierw aktualizacje, potem cała reszta. Pełny upgrade i restart, zanim powstanie choćby jedna VM. Przenoszenie VM-ek później tylko po to, żeby załatać hosta, to ból, którego można uniknąć.

Tworzenie klastra. Gdy oba węzły są zainstalowane i zaktualizowane, utworzenie klastra to dwa polecenia — pvecm create na pierwszym węźle, pvecm add na drugim. Od tej chwili jeden interfejs webowy zarządza oboma maszynami.

Problem kworum. Klaster dwuwęzłowy ma wbudowaną wadę: gdy jeden węzeł padnie, ocalały nie ma większości i odmawia operacji klastrowych. Rozwiązaniem jest QDevice — trzeci głos, który nie musi być pełnoprawnym węzłem Proxmoxa. U mnie corosync-qnetd działa na Raspberry Pi 3, pełniącym rolę rozjemcy. Nie robi nic poza głosowaniem — i to wystarcza, by klaster działał, gdy węzeł jest niedostępny.

# Na klastrze, po skonfigurowaniu Pi:
pvecm qdevice setup <pi-ip>

Backupy: Proxmox Backup Server

Homelab bez backupów to bomba zegarowa z interfejsem webowym. Zanim wdrożyłem cokolwiek, czego byłoby mi szkoda, postawiłem Proxmox Backup Server (PBS) — jako VM (VMID 110) na samym klastrze.

Tak, uruchamianie serwera backupów na infrastrukturze, którą on sam zabezpiecza, to kompromis. Najważniejsze: właściwy datastore leży na osobnym storage, nie na dyskach klastra. Gdy węzeł padnie, backupy przetrwają; sam PBS można przeinstalować w dwadzieścia minut.

Dlaczego PBS zamiast zwykłych backupów vzdump do katalogu?

  • Deduplikacja — dziesiątki snapshotów zajmują ułamek surowej przestrzeni, bo niezmienione fragmenty są przechowywane raz
  • Backupy przyrostowe — po pierwszym pełnym backupie przesyłane są tylko zmienione bloki. Nocne backupy kończą się w minuty
  • Zadania weryfikacji — PBS okresowo sprawdza zapisane fragmenty względem sum kontrolnych, więc korupcję znajduję zanim potrzebny będzie restore
  • Polityki retencji — keep-last, keep-daily, keep-weekly skonfigurowane raz, egzekwowane automatycznie

Podpięcie jest proste: dodaj PBS jako storage w datacenter Proxmoxa, utwórz zadanie backupu obejmujące wszystkie VM-ki, zaplanuj co noc, gotowe.

Zasada, której się trzymam: żaden workload nie istnieje, dopóki nie obejmuje go zadanie backupu. Nowa VM, nowy kontener — trafia do harmonogramu backupów tego samego dnia, w którym powstał.

Co z tego wynika

Na koniec kroku pierwszego fundament wygląda tak:

  • Dwa węzły Proxmoxa, załatane, statyczne IP, w klastrze
  • QDevice pilnujące uczciwego kworum
  • PBS z nocnymi, deduplikowanymi backupami i weryfikacją

Jeszcze żadnych usług. Żadnych kontenerów. Nic ekscytującego. Ale wszystko, co przyjdzie później — a przyjdzie sporo — siedzi na tym fundamencie i jest backupowane od pierwszego dnia.

O to właśnie chodzi w kroku pierwszym: zbudować nudną warstwę solidnie, żeby nigdy więcej nie trzeba było o niej myśleć.
