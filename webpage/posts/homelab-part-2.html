Homelab Part 2: The Raspberry Pi Tiebreaker and a 40Gbps Thunderbolt Cluster Link

How a €30 Pi keeps my cluster honest, and how two laptops got a 40Gbps interconnect without a single expensive switch.

In Part 1 I covered installing Proxmox on two Dell Precision 3561 nodes and getting backups running. This post digs into two things I only mentioned in passing: the Raspberry Pi 3 QDevice and the dual network setup — 1Gb Ethernet to the LAN plus a direct 40Gbps Thunderbolt 4 link between the nodes.

Part A: The Raspberry Pi That Does Almost Nothing

The two-node problem, properly explained

Proxmox clustering runs on Corosync, and Corosync makes decisions by majority vote. Every node gets one vote, and the cluster only operates when more than half the votes are present — that's quorum.

Do the math on a two-node cluster:

  • Both nodes up: 2 of 2 votes → quorum ✔
  • One node down: 1 of 2 votes → no majority → quorum lost ✘

So the moment one node fails — the exact scenario a cluster is supposed to handle — the surviving node locks itself down. No starting VMs, no config changes. The cluster protects itself against split-brain by refusing to act, which is correct behavior and completely useless to you at 2 AM.

Enter the QDevice

The fix doesn't require a third full Proxmox node. Corosync supports an external QDevice: a lightweight daemon (corosync-qnetd) running on any Linux box that provides a third vote without being part of the cluster. It runs no VMs, holds no data, and participates in nothing except arbitration.

This is the perfect job for a Raspberry Pi 3 gathering dust in a drawer:

  • Draws about 2-3 watts
  • Completely passive — it literally just answers "who should have quorum?"
  • If the Pi itself dies while both nodes are healthy, nothing happens (2 of 3 votes is still a majority)

Setup

On the Pi (running Raspberry Pi OS / Debian):

apt install corosync-qnetd

On the Proxmox nodes:

apt install corosync-qdevice   # on both nodes
pvecm qdevice setup <pi-ip>    # once, from either node

Verify:

pvecm status
# Expected: 3 total votes, "Qdevice" listed, quorum 2

Now any single failure — node A, node B, or the Pi itself — leaves 2 of 3 votes standing, and the cluster keeps working. Three points of failure, tolerance for any one of them, total extra cost: one Pi and a power brick.

One practical note: give the Pi a static IP and treat it like infrastructure, not a toy. A QDevice that wanders off DHCP is worse than no QDevice, because you'll debug quorum weirdness before you think to check the Pi.

Part B: Two Networks, Two Jobs

The nodes have two completely separate network paths, and the design principle is simple: cluster traffic and user traffic should never compete.

Link 1: 1Gb Ethernet — the front door

Both nodes connect to the LAN over standard gigabit Ethernet (192.168.1.100 and .101). This carries:

  • Web UI and SSH management access
  • VM traffic to the rest of the network and the internet
  • Backup traffic to storage
  • Corosync heartbeats

1GbE is plenty for all of this. Management is kilobytes, and even backup jobs rarely saturate the line thanks to PBS incremental transfers.

Link 2: Thunderbolt 4 — the 40Gbps backbone

Here's the trick that makes laptop-based clustering genuinely good: both Precision 3561s have Thunderbolt 4 ports, and Thunderbolt supports IP networking over a direct cable. Plug one TB4 cable between the two nodes, and Linux exposes it as a network interface via the thunderbolt-net kernel module.

That's a 40Gbps point-to-point link for the price of one cable. No 10/25/40GbE NICs, no switch, no SFP+ modules — hardware that would cost more than the laptops themselves.

The interface shows up (typically as thunderbolt0 / en0 style naming depending on setup), and you configure it like any other NIC — just on its own private subnet, e.g.:

pve-01: 10.0.0.1/30  (thunderbolt)
pve-02: 10.0.0.2/30  (thunderbolt)

A /30 is all you need — two hosts, point to point, nothing else will ever live there.

What runs over the Thunderbolt link

Everything heavy that happens between the nodes:

  • Live migration. Moving a running VM means shipping its entire RAM contents across. Over 1GbE, a VM with 8GB of RAM takes over a minute of transfer; over the TB4 link it's seconds. Migration goes from "schedule it" to "just do it."
  • Bulk transfers between nodes — moving disks, templates, restore data from one node to the other.

In the Proxmox datacenter options, you set the migration network to the Thunderbolt subnet, and Proxmox routes all migration traffic there automatically:

Datacenter → Options → Migration Settings → Network: 10.0.0.0/30

What deliberately doesn't

Corosync stays on Ethernet. This sounds backwards — why not put cluster communication on the fastest link? Because Corosync doesn't care about bandwidth; it cares about latency and reliability, and it needs a network that both nodes and the QDevice can reach. The Pi has no Thunderbolt port. Cluster heartbeats, quorum votes, and QDevice arbitration all live on the 1GbE LAN where all three participants exist.

The split, summarized:

TrafficLinkWhy
Management, VM traffic, backups1GbEReaches everything, plenty fast
Corosync + QDevice1GbEPi must be reachable; latency matters, not bandwidth
Live migration, node-to-node bulkTB4 (40Gbps)RAM-sized transfers, isolated from everything else

Caveats worth knowing

  • Real-world throughput won't hit 40Gbps. The thunderbolt-net path has CPU overhead, and actual numbers land well below the line rate — but still several times faster than anything you'd get from affordable Ethernet gear.
  • It doesn't survive weirdness like Ethernet does. Cable reseats or suspend/resume can drop the link; a systemd unit or udev rule to reconfigure the interface on hotplug saves headaches.
  • Two nodes only. Thunderbolt networking is point-to-point. This design works precisely because the cluster is a pair — a third node would need a different answer.

The Shape of It

Three boxes, two networks:

  • Two Proxmox nodes joined by a private 40Gbps Thunderbolt backbone for the heavy lifting
  • Both nodes plus a Raspberry Pi on the 1GbE LAN, where quorum lives
  • Any one of the three can die and the cluster shrugs

The total cost of the resilience layer — the Pi and the TB4 cable — is less than a nice dinner. That's the homelab sweet spot: enterprise concepts, pocket-money hardware.

Part 3 will build upward from here. The foundation is done.

Homelab Teil 2: Der Raspberry Pi als Zünglein an der Waage und ein 40-Gbps-Thunderbolt-Clusterlink

Wie ein 30-€-Pi meinen Cluster ehrlich hält — und wie zwei Laptops zu einem 40-Gbps-Interconnect kamen, ganz ohne teuren Switch.

In Teil 1 ging es um die Proxmox-Installation auf zwei Dell Precision 3561 und laufende Backups. Dieser Beitrag vertieft zwei Dinge, die ich nur am Rande erwähnt habe: das Raspberry Pi 3 QDevice und das doppelte Netzwerk-Setup — 1-Gb-Ethernet ins LAN plus ein direkter 40-Gbps-Thunderbolt-4-Link zwischen den Knoten.

Teil A: Der Raspberry Pi, der fast nichts tut

Das Zwei-Knoten-Problem, richtig erklärt

Proxmox-Clustering läuft auf Corosync, und Corosync entscheidet per Mehrheitsvotum. Jeder Knoten hat eine Stimme, und der Cluster arbeitet nur, wenn mehr als die Hälfte der Stimmen anwesend ist — das ist das Quorum.

Rechnen wir für einen Zwei-Knoten-Cluster nach:

  • Beide Knoten oben: 2 von 2 Stimmen → Quorum ✔
  • Ein Knoten unten: 1 von 2 Stimmen → keine Mehrheit → Quorum verloren ✘

In dem Moment, in dem ein Knoten ausfällt — genau das Szenario, für das ein Cluster da ist — sperrt sich der überlebende Knoten selbst. Keine VMs starten, keine Konfigurationsänderungen. Der Cluster schützt sich vor Split-Brain, indem er nichts mehr tut. Korrekt — und um 2 Uhr nachts völlig nutzlos.

Auftritt: das QDevice

Die Lösung braucht keinen dritten vollwertigen Proxmox-Knoten. Corosync unterstützt ein externes QDevice: ein leichtgewichtiger Daemon (corosync-qnetd) auf einer beliebigen Linux-Kiste, der eine dritte Stimme liefert, ohne Teil des Clusters zu sein. Er betreibt keine VMs, hält keine Daten und beteiligt sich an nichts außer der Schiedsrichterrolle.

Der perfekte Job für einen Raspberry Pi 3, der in der Schublade verstaubt:

  • Verbraucht etwa 2-3 Watt
  • Völlig passiv — er beantwortet buchstäblich nur „wer soll das Quorum haben?"
  • Stirbt der Pi selbst, während beide Knoten gesund sind, passiert nichts (2 von 3 Stimmen sind immer noch eine Mehrheit)

Einrichtung

Auf dem Pi (mit Raspberry Pi OS / Debian):

apt install corosync-qnetd

Auf den Proxmox-Knoten:

apt install corosync-qdevice   # auf beiden Knoten
pvecm qdevice setup <pi-ip>    # einmal, von einem der Knoten

Prüfen:

pvecm status
# Erwartet: 3 Stimmen gesamt, "Qdevice" gelistet, Quorum 2

Jetzt lässt jeder einzelne Ausfall — Knoten A, Knoten B oder der Pi selbst — 2 von 3 Stimmen stehen, und der Cluster arbeitet weiter. Drei mögliche Ausfallpunkte, Toleranz für jeden einzelnen, Gesamtkosten: ein Pi und ein Netzteil.

Ein praktischer Hinweis: Gib dem Pi eine statische IP und behandle ihn wie Infrastruktur, nicht wie Spielzeug. Ein QDevice, das per DHCP herumwandert, ist schlimmer als keins — man debuggt Quorum-Merkwürdigkeiten, bevor man auf die Idee kommt, den Pi zu prüfen.

Teil B: Zwei Netzwerke, zwei Aufgaben

Die Knoten haben zwei komplett getrennte Netzwerkpfade, und das Designprinzip ist simpel: Cluster-Traffic und Nutzer-Traffic dürfen nie konkurrieren.

Link 1: 1-Gb-Ethernet — die Eingangstür

Beide Knoten hängen über Standard-Gigabit-Ethernet im LAN (192.168.1.100 und .101). Darüber laufen:

  • Web-UI- und SSH-Verwaltungszugriff
  • VM-Traffic ins restliche Netz und ins Internet
  • Backup-Traffic zum Storage
  • Corosync-Heartbeats

1GbE reicht dafür locker. Verwaltung sind Kilobytes, und selbst Backup-Jobs sättigen die Leitung dank inkrementeller PBS-Transfers selten.

Link 2: Thunderbolt 4 — das 40-Gbps-Rückgrat

Hier kommt der Trick, der Laptop-Clustering richtig gut macht: Beide Precision 3561 haben Thunderbolt-4-Ports, und Thunderbolt unterstützt IP-Netzwerk über ein Direktkabel. Ein TB4-Kabel zwischen die Knoten gesteckt, und Linux stellt es über das Kernelmodul thunderbolt-net als Netzwerkschnittstelle bereit.

Das ist ein 40-Gbps-Punkt-zu-Punkt-Link zum Preis eines Kabels. Keine 10/25/40GbE-NICs, kein Switch, keine SFP+-Module — Hardware, die mehr kosten würde als die Laptops selbst.

Die Schnittstelle taucht auf (je nach Setup als thunderbolt0 / en0-artige Benennung) und wird wie jede andere NIC konfiguriert — nur auf einem eigenen privaten Subnetz, z. B.:

pve-01: 10.0.0.1/30  (thunderbolt)
pve-02: 10.0.0.2/30  (thunderbolt)

Ein /30 genügt völlig — zwei Hosts, Punkt zu Punkt, dort wird nie etwas anderes wohnen.

Was über den Thunderbolt-Link läuft

Alles Schwere, das zwischen den Knoten passiert:

  • Live-Migration. Eine laufende VM zu verschieben heißt, ihren gesamten RAM-Inhalt zu übertragen. Über 1GbE braucht eine VM mit 8 GB RAM über eine Minute; über den TB4-Link sind es Sekunden. Migration wird von „einplanen" zu „einfach machen".
  • Bulk-Transfers zwischen Knoten — Disks, Templates, Restore-Daten von einem Knoten zum anderen.

In den Datacenter-Optionen von Proxmox setzt man das Migrationsnetzwerk auf das Thunderbolt-Subnetz, und Proxmox leitet allen Migrationsverkehr automatisch dorthin:

Datacenter → Options → Migration Settings → Network: 10.0.0.0/30

Was bewusst nicht darüber läuft

Corosync bleibt auf Ethernet. Das klingt verkehrt — warum die Cluster-Kommunikation nicht auf den schnellsten Link legen? Weil Corosync Bandbreite egal ist; es geht um Latenz und Zuverlässigkeit, und es braucht ein Netz, das beide Knoten und das QDevice erreichen. Der Pi hat keinen Thunderbolt-Port. Cluster-Heartbeats, Quorum-Abstimmungen und QDevice-Arbitrierung leben alle im 1GbE-LAN, wo alle drei Teilnehmer existieren.

Die Aufteilung, zusammengefasst:

TrafficLinkWarum
Verwaltung, VM-Traffic, Backups1GbEErreicht alles, schnell genug
Corosync + QDevice1GbEPi muss erreichbar sein; Latenz zählt, nicht Bandbreite
Live-Migration, Bulk zwischen KnotenTB4 (40Gbps)RAM-große Transfers, von allem anderen isoliert

Wissenswerte Einschränkungen

  • Realer Durchsatz erreicht keine 40 Gbps. Der thunderbolt-net-Pfad hat CPU-Overhead, und echte Zahlen liegen deutlich unter der Leitungsrate — aber immer noch um ein Mehrfaches über allem, was bezahlbare Ethernet-Hardware liefert.
  • Er übersteht Störungen nicht so wie Ethernet. Kabel-Neustecken oder Suspend/Resume können den Link trennen; eine systemd-Unit oder udev-Regel, die die Schnittstelle bei Hotplug neu konfiguriert, erspart Kopfschmerzen.
  • Nur zwei Knoten. Thunderbolt-Netzwerk ist Punkt zu Punkt. Dieses Design funktioniert genau deshalb, weil der Cluster ein Paar ist — ein dritter Knoten bräuchte eine andere Antwort.

Das Gesamtbild

Drei Kisten, zwei Netzwerke:

  • Zwei Proxmox-Knoten, verbunden durch ein privates 40-Gbps-Thunderbolt-Rückgrat für die schwere Arbeit
  • Beide Knoten plus ein Raspberry Pi im 1GbE-LAN, wo das Quorum lebt
  • Jede der drei Kisten darf sterben, und der Cluster zuckt mit den Schultern

Die Gesamtkosten der Resilienz-Schicht — Pi und TB4-Kabel — liegen unter einem guten Abendessen. Das ist der Homelab-Sweetspot: Enterprise-Konzepte, Taschengeld-Hardware.

Teil 3 baut von hier nach oben. Das Fundament steht.

Homelab część 2: Raspberry Pi jako rozjemca i łącze klastrowe Thunderbolt 40 Gb/s

Jak Pi za 30 € pilnuje uczciwości klastra i jak dwa laptopy dostały łącze 40 Gb/s bez ani jednego drogiego switcha.

W części 1 opisałem instalację Proxmoxa na dwóch węzłach Dell Precision 3561 i uruchomienie backupów. Ten wpis zagłębia się w dwie rzeczy, o których wspomniałem tylko mimochodem: QDevice na Raspberry Pi 3 i podwójną konfigurację sieci — 1 Gb Ethernet do LAN-u plus bezpośrednie łącze Thunderbolt 4 40 Gb/s między węzłami.

Część A: Raspberry Pi, które nie robi prawie nic

Problem dwóch węzłów, porządnie wyjaśniony

Klastrowanie Proxmoxa opiera się na Corosyncu, a Corosync podejmuje decyzje większością głosów. Każdy węzeł ma jeden głos, a klaster działa tylko wtedy, gdy obecna jest ponad połowa głosów — to jest kworum.

Policzmy dla klastra dwuwęzłowego:

  • Oba węzły działają: 2 z 2 głosów → kworum ✔
  • Jeden węzeł pada: 1 z 2 głosów → brak większości → kworum utracone ✘

Czyli w momencie awarii jednego węzła — dokładnie w scenariuszu, do którego klaster jest stworzony — ocalały węzeł sam się blokuje. Żadnego startowania VM-ek, żadnych zmian konfiguracji. Klaster chroni się przed split-brainem, odmawiając działania. Zachowanie poprawne — i kompletnie bezużyteczne o 2 w nocy.

Wchodzi QDevice

Rozwiązanie nie wymaga trzeciego pełnoprawnego węzła Proxmoxa. Corosync obsługuje zewnętrzny QDevice: lekki daemon (corosync-qnetd) działający na dowolnej linuksowej maszynie, który dostarcza trzeci głos, nie będąc częścią klastra. Nie uruchamia VM-ek, nie trzyma danych i nie uczestniczy w niczym poza arbitrażem.

To idealna robota dla Raspberry Pi 3 kurzącego się w szufladzie:

  • Pobiera około 2-3 watów
  • Całkowicie pasywny — dosłownie tylko odpowiada na pytanie „kto ma mieć kworum?"
  • Jeśli samo Pi padnie, gdy oba węzły są zdrowe, nic się nie dzieje (2 z 3 głosów to nadal większość)

Konfiguracja

Na Pi (z Raspberry Pi OS / Debianem):

apt install corosync-qnetd

Na węzłach Proxmoxa:

apt install corosync-qdevice   # na obu węzłach
pvecm qdevice setup <pi-ip>    # raz, z dowolnego węzła

Weryfikacja:

pvecm status
# Oczekiwane: 3 głosy łącznie, "Qdevice" na liście, kworum 2

Teraz każda pojedyncza awaria — węzeł A, węzeł B albo samo Pi — zostawia 2 z 3 głosów, a klaster działa dalej. Trzy punkty awarii, tolerancja na każdy z nich, całkowity dodatkowy koszt: jedno Pi i zasilacz.

Jedna praktyczna uwaga: daj Pi statyczne IP i traktuj je jak infrastrukturę, nie zabawkę. QDevice błądzące po DHCP jest gorsze niż brak QDevice, bo będziesz debugować dziwactwa kworum, zanim przyjdzie ci do głowy sprawdzić Pi.

Część B: Dwie sieci, dwa zadania

Węzły mają dwie całkowicie oddzielne ścieżki sieciowe, a zasada projektowa jest prosta: ruch klastrowy i ruch użytkowników nigdy nie powinny konkurować.

Łącze 1: Ethernet 1 Gb — drzwi frontowe

Oba węzły łączą się z LAN-em przez standardowy gigabitowy Ethernet (192.168.1.100 i .101). Tędy idzie:

  • Dostęp zarządczy przez web UI i SSH
  • Ruch VM-ek do reszty sieci i internetu
  • Ruch backupowy do storage
  • Heartbeaty Corosynca

1GbE w zupełności wystarcza. Zarządzanie to kilobajty, a nawet zadania backupu rzadko nasycają łącze dzięki przyrostowym transferom PBS.

Łącze 2: Thunderbolt 4 — kręgosłup 40 Gb/s

Oto trik, który czyni klastrowanie na laptopach naprawdę dobrym: oba Precision 3561 mają porty Thunderbolt 4, a Thunderbolt obsługuje sieć IP po bezpośrednim kablu. Wepnij jeden kabel TB4 między węzły, a Linux wystawi go jako interfejs sieciowy przez moduł jądra thunderbolt-net.

To łącze punkt-punkt 40 Gb/s w cenie jednego kabla. Żadnych kart 10/25/40GbE, żadnego switcha, żadnych modułów SFP+ — sprzętu, który kosztowałby więcej niż same laptopy.

Interfejs pojawia się w systemie (zwykle jako thunderbolt0 / nazwa w stylu en0, zależnie od konfiguracji) i konfiguruje się go jak każdą inną kartę — tylko we własnej prywatnej podsieci, np.:

pve-01: 10.0.0.1/30  (thunderbolt)
pve-02: 10.0.0.2/30  (thunderbolt)

/30 to wszystko, czego potrzeba — dwa hosty, punkt-punkt, nic innego tam nigdy nie zamieszka.

Co biegnie po łączu Thunderbolt

Wszystko ciężkie, co dzieje się między węzłami:

  • Migracja na żywo. Przeniesienie działającej VM oznacza przesłanie całej zawartości jej RAM-u. Po 1GbE maszyna z 8 GB RAM potrzebuje ponad minuty transferu; po łączu TB4 to sekundy. Migracja zmienia się z „zaplanuj to" w „po prostu zrób".
  • Masowe transfery między węzłami — przenoszenie dysków, szablonów, danych restore z jednego węzła na drugi.

W opcjach datacenter Proxmoxa ustawia się sieć migracji na podsieć Thunderbolt i Proxmox automatycznie kieruje tam cały ruch migracyjny:

Datacenter → Options → Migration Settings → Network: 10.0.0.0/30

Co celowo nie biegnie

Corosync zostaje na Ethernecie. Brzmi to na odwrót — czemu nie puścić komunikacji klastra po najszybszym łączu? Bo Corosynca nie obchodzi przepustowość; obchodzi go opóźnienie i niezawodność, a potrzebuje sieci, do której dotrą oba węzły i QDevice. Pi nie ma portu Thunderbolt. Heartbeaty klastra, głosowania kworum i arbitraż QDevice żyją w LAN-ie 1GbE, gdzie istnieją wszyscy trzej uczestnicy.

Podział w skrócie:

RuchŁączeDlaczego
Zarządzanie, ruch VM, backupy1GbEDociera wszędzie, wystarczająco szybkie
Corosync + QDevice1GbEPi musi być osiągalne; liczy się opóźnienie, nie przepustowość
Migracja na żywo, transfery masoweTB4 (40Gb/s)Transfery wielkości RAM-u, odizolowane od reszty

Zastrzeżenia warte znajomości

  • Rzeczywista przepustowość nie sięgnie 40 Gb/s. Ścieżka thunderbolt-net ma narzut CPU i realne liczby lądują wyraźnie poniżej nominalnej prędkości — ale wciąż kilkukrotnie szybciej niż cokolwiek z przystępnego cenowo sprzętu Ethernet.
  • Nie znosi dziwactw tak jak Ethernet. Przepięcie kabla albo suspend/resume potrafi zerwać łącze; unit systemd lub reguła udev rekonfigurująca interfejs przy hotplugu oszczędza bólu głowy.
  • Tylko dwa węzły. Sieć Thunderbolt jest punkt-punkt. Ten projekt działa właśnie dlatego, że klaster to para — trzeci węzeł wymagałby innej odpowiedzi.

Całokształt

Trzy pudełka, dwie sieci:

  • Dwa węzły Proxmoxa połączone prywatnym kręgosłupem Thunderbolt 40 Gb/s do ciężkiej roboty
  • Oba węzły plus Raspberry Pi w LAN-ie 1GbE, gdzie mieszka kworum
  • Każde z trzech może umrzeć, a klaster wzrusza ramionami

Łączny koszt warstwy odporności — Pi i kabel TB4 — to mniej niż dobra kolacja. To jest właśnie słodki punkt homelabu: koncepty klasy enterprise, sprzęt za kieszonkowe.

Część 3 będzie budować w górę. Fundament gotowy.
